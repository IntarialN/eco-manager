# Git Workflow проекта

Цель — поддерживать предсказуемый цикл разработки и одинаковые правила для всей команды.

## 1. Ветки

- `main` — стабильная ветка, отражает состояние, готовое к выкладке на production. Мержим только через pull request из `develop` после прохождения всех проверок.
- `develop` — основная ветка разработки. В неё попадают изменения после ревью и тестов. На базе `develop` создаём feature-ветки.
- `feature/<topic>` — рабочие ветки для задач. Пример: `feature/auth-registration`. Создаются от `develop`, после завершения — merge request обратно в `develop`.
- `release/<version>` (опционально) — подготовка релизов. Создаём, когда нужно стабилизировать версию перед выкладкой. После релиза мержим в `main` и `develop`.
- `hotfix/<issue>` — экстренные исправления на production. Создаются от `main`, после завершения мержим в `main` и `develop`.

## 2. Процесс

1. **Старт задачи**: ветка `feature/<topic>` от `develop`.
2. **Коммиты**: атомарные, с префиксами по типу (`docs:`, `feat:`, `fix:`). Желательно следовать Conventional Commits.
3. **Проверки**: локально запускаем lint/test. CI на PR повторяет проверки + сборку.
4. **Review & Merge**: PR в `develop`, минимум один ревьюер. Сквош или merge — по договорённости, но история должна оставаться читаемой.
5. **Релиз**: создаём `release/<version>`, проводим финальное тестирование. После релиза тэг `vX.Y.Z` на `main`.
6. **Хотфиксы**: при критических исправлениях работаем от `main`, после мёржа обязательно синхронизируем `develop`.

## 3. Настройка по текущему состоянию

- Созданы ветки `main`, `develop`, `feature/docs-documentation-setup` (как текущая рабочая ветка).
- По готовности документации `feature/docs-documentation-setup` мержим в `develop`, затем в `main` через PR.
- Установить GitHub default branch → `main`, защитить ветки `main` и `develop` (обязательный PR, CI, код-ревью).

## 4. Практические рекомендации

- Перед началом работы `git fetch` и `git pull --rebase origin develop`.
- Не держать feature-ветки дольше 3–4 дней без синхронизации.
- Для больших задач разбивать на несколько PR (документация, бекенд, фронтенд).
- В `docs/completed/` фиксировать номер ветки и коммита, чтобы легко отследить изменение.
